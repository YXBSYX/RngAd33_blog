信息收集

展开攻击之前的必要步骤；

源码泄露

  1. 开发注释未及时删除,按F12可以查看源码获取信息；

  2. 有些网页设置了js前台绕过无法按F12，在网址前端添加view-source:或者前往浏览器设置关闭javascript即可绕过；

  3. 对于采用PHP语言且缺乏防护的网页，通过以下访问手段可以获取到网页PHP源代码；

  4. 版本控制泄露

敏感数据泄露

  1. robots后台泄露
    robots.txt规定了用户能够访问网页的范围；
    缺乏防护时，访问robots.txt可以获取到网页关键路径等信息；

  2. 探针泄露
    开发完成后，未及时删除的探针可能导致信息泄露；
    记得删除探针，或者将探针命名为复杂不易猜的名字。

  3. cookie泄露
    cookie的本质是一个文档，它会记录用户的访问信息；
    如果没有对cookie设置HttpOnly属性，一旦被截获就会造成信息泄露；

  4. 数据库恶意下载
    mdb文件是早期asp + access架构的数据库文件，文件泄露相当于数据库被脱裤了；
    与此同时，备份的sql文件也可能会泄露敏感信息；

暴力破解

想要实现网页爆破，你需要学会使用Burpsuite

  1. 纯数字情形
    对于纯数字组合的信息，可以使用Burpsuite设置区间构造字典实施爆破。爆破完成后，通过返回长度可以很容易地确认密码，随后查看响应内容获取信息。

  2. 一般情形
    如果爆破内容包含了字母以及特殊字符，也可以采用上面的方法，但是比较费时间。最好调用自己提前准备好的字典，后面的操作方法同上。

  3. 随机数情形
    如果要爆破的信息是用随机数种子动态生成的，就不要头铁爆破了（这里用到了伪随机数漏洞，详见下文）。先在本地搭建好PHP环境或Linux环境，然后调用靶场的随机数种子生成随机数，最后传参。

命令执行

命令执行，需要严格的过滤。攻击者需要想方设法绕过重重过滤。

敏感词汇过滤

  1. 文件名/扩展名过滤
    使用*或?替换掉部分文件名和扩展名，前者可以替换掉一个字符，后者可以一次替换掉多个字符;
    或者在词汇中间插入’’或者\绕过检测；

  2. 函数名过滤
    当system被过滤时，使用echo `command`;执行系统命令；
    如果echo被过滤，使用nl命令；

  3. 拼接法
    通过构造与调用参数拼接敏感词汇，也能够绕过检测；

  4. 包含法
    利用到了文件包含漏洞当中的filter包含（还挺好用）；

  5. 文件名/扩展名过滤plus版
    在第一个的基础上做了升级。比如要过滤关键词tac，凡是传入的参数当中按顺序出现了’t’’a’’c’，参数就会被过滤；
    此时前面的方法就不好用了，需要在省略部分字符的前提下加上路径否则无法匹配命令；

敏感字符过滤

  1. 分号;过滤
    使用%0a代替分号；

  2. 空格过滤
    在echo没有被过滤的情况下，使用$IFS（echo适用）或${IFS}代替空格；
    如果echo被过滤，使用%09代替空格（%09是Tab键的url编码）；
    如果$或{}或IFS被过滤，可以获取并调用php环境中与空格等效的标识符；

  3. 问号?过滤
    使用不含问号的语句构造一句话；

数字、字母过滤

如果遇到了所有字母和数字都被过滤的极端情况，使用以下几种方法：

  1. php异或
    将系统命令异或处理后传入（需要保证密文和密钥均不包含字母和数字）；
  2. php或
    与异或相似，操作是差不多的。

产生原因

  1. 可控参数
    Web应用系统的许多功能需要前后端交互，如果没有对参数进行安全测试，就可能会出现注入漏洞；

  2. 参数未过滤或者过滤不严谨
    没有对交互过程进行威胁内容过滤，很可能造成注入漏洞；

  3. 可控参数可与数据库交互
    与数据库交互，这种行为本身就可能存在注入漏洞；

注入的危害

  1. 非法读取、篡改、添加、删减数据库中的数据；
  2. 规避认证；
  3. 通过修改数据库，篡改网页内容；
  4. 私自盗取用户信息、添加或删除账号；
  5. 注入木马程序；

防护策略

  1. 过滤单引号；
  2. 对所有来自外部的用户输入进行安全检查；
  3. 使用类型安全的参数编码机制构造动态SQL语句；
  4. 尽量不使用动态拼接的SQL，使用参数化的SQL；
  5. 采用一系列安全措施加强的SQL服务器的配置与链接；
SQL注入攻击一般通过目标服务器80端口进行，产生的请求信息与常规网页访问没有什么区别，所以一般的防火墙不会进行警告或拦截。

文件包含漏洞

使用各类文件包含指令，对目标实施攻击；

包含种类

  1. data包含
    用于执行一句话木马，查找并获取flag。
  2. filter包含
    用于获取网页PHP源代码，但在不知道flag在哪时不好用。
  3. http日志包含
    nginx默认日志路径。配合burpsuite使用，与data包含功能相似，伟大无需多言。 包含前，拦截http请求头，在ua信息末尾插入一句话木马（一般用来传参）后放行请求头；
    传参后，在日志包含口令后面添加系统命令，随后可以找到flag；
  4. SSH日志包含
    与http日志包含相似，利用恶意用户名注入一句话木马； 请注意：日志文件会记录最为原始的URL请求，在浏览器地址栏中输入的地址会被URL编码，
    通过CURl或者Burp改包绕过编码。
  5. session包含
    php将用户session以文件的形式保存在主机中，通过php.ini文件中的session.save_path字段可以设置具体的存储位置，通过phpinfo()也可以查询到；
    一些可能的session文件存放路径： 另一种思路：通过条件竞争实现session包含： 利用这个脚本，随便上传一个文件（文件不能太大，不然会502报错），实际上是通过上传PHP_SESSION_UPLOAD_PROGRESS
    并访问/tmp/sess_xxx，利用burpsuite抓包后不断发送访问请求，实现文件包含；
    在Cookie里设置参数PHPSESSID=exp，php将会在服务器上创建一个文件：/tmp/sess_exp； 但是对于默认配置session.upload_progress.cleanup = on，文件上传后session文件内容会立即被清空，
    对此，我们需要通过条件竞争，在服务器还未来得及删除我们上传的session文件内容时包含到该文件，实现恶意代码的命令执行。

IIS解析漏洞

针对IIS6.0，其在解析文件时存在以下两种漏洞：

  1. 目录解析漏洞
    *.asa或*.asp格式的文件夹下的任意文件均能被当作asp文件解析；

  2. 文件名解析漏洞
    *.asp;1.jpg文件会被当做asp文件执行；

  3. 畸形解析漏洞
    微软发布了IIS7.0修补了IIS6.0的解析漏洞，没想到IIS7.0爆出更严重的畸形解析漏洞；

防护策略

XSS跨站脚本漏洞

过滤验证很重要，但如果只设置javascript前端验证，忽略了后端验证，就会留下漏洞。

比如说，我们要在网页上输入一段信息，javascript验证可以在提交数据之前过滤掉敏感字符，这样主要是为了防止用户输入错误。

然而，绕过javascript验证的思路也非常简单：先提交合法数据，再拦截上传的http(s)请求并将原先输入的合法数据修改为敏感数据，最后上传篡改后的http(s)请求，就能够成功提敏感数据，造成XSS漏洞。

所以，设置服务器端验证是非常重要的攻击防护手段，每一位web开发人员都应当具有这样的意识。

CSRF漏洞

跨站请求伪造漏洞，黑客通过盗用身份建立会话，进行一系列非法操作，或者欺骗用户访问带有木马的url链接。传说中的钓鱼链接就是这样来的。

防护策略:

  1. 要求用户进行二次确认；
  2. 阻止不明外域的访问；
  3. 提交时要求附加只有在本域才能获取的信息；
  4. 增加token（令牌）验证（token需要随机生成，值足够大，用后需销毁）；
  5. 增加验证码；
请注意：当网站同时存在XSS漏洞和CSRF漏洞时，token验证机制将会失效！

逻辑错误漏洞

当一个系统的功能太多的时候，难免会出现由开发人员思路错误或思考不全面导致的程序逻辑问题，导致了逻辑漏洞；

漏洞挖掘

发现网页功能->了解业务流程->拦截、篡改http(s)请求

SSRF漏洞

如果服务端提供了从其他服务器应用获取数据的功能且没有对目标地址做过滤与限制，黑客可以通过伪造服务端请求对其他服务器实施攻击，一般用于攻击与外界隔离的内部服务器；

利用方式

  1. 探测内网信息；
  2. 窃取敏感内容；
  3. 作为跳板，攻击其他网站或应用；
